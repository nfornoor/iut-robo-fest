# System Architecture — Office Device Monitoring Platform

**Project:** Lights, Fans, Discord: The Boss's Big Idea
**Scope:** Web dashboard + Discord bot sharing a single backend, monitoring 15 simulated devices across 3 rooms.

---

## 1. Overview

The system simulates an office of 3 rooms (Drawing Room, Work Room 1, Work Room 2), each with 2 fans and 3 lights (15 devices total). A backend service owns the single source of truth for device state and energy usage. A web dashboard and a Discord bot both read from — and in the dashboard's case, can also write to (manual toggle) — this same backend, so they never disagree with each other.

There is no physical hardware. Device state is generated by a **simulator** running inside the backend, plus optional manual overrides for demo purposes.

## 2. High-Level Architecture

```
                         ┌─────────────────────────┐
                         │   Device Simulator        │
                         │   (background task)       │
                         │  - periodic random toggle  │
                         │  - manual override hook    │
                         └───────────┬────────────────┘
                                     │ updates
                                     ▼
                         ┌─────────────────────────┐
                         │   Backend (FastAPI)       │
                         │  - In-memory device store │
                         │  - Energy accumulator     │
                         │  - Alert engine            │
                         │  - REST API                │
                         │  - WebSocket broadcaster    │
                         └──────┬──────────────┬─────┘
                                │              │
                 REST + WS      │              │  REST (poll on command)
                                ▼              ▼
                  ┌───────────────────┐   ┌───────────────────┐
                  │   Web Dashboard     │   │   Discord Bot        │
                  │   (Next.js)          │   │   (discord.py)        │
                  │  - live status panel │   │  - !status            │
                  │  - power meter        │   │  - !room <name>       │
                  │  - alerts panel        │   │  - !usage              │
                  │  - office layout view  │   │  - LLM-phrased replies │
                  └──────────┬──────────┘   └──────────┬──────────┘
                             │                          │
                             ▼                          ▼
                          Boss / User views on browser or in Discord
```

**Key principle:** the arrows only flow *out* of the backend to the two interfaces (plus manual toggle back in from the dashboard). Neither interface talks to the other directly, and neither keeps its own copy of state.

## 3. Components

### 3.1 Device Simulator
Runs as an `asyncio` background task inside the backend process (not a separate service, to keep the "single source of truth" simple).

Responsibilities:
- Every N seconds (e.g., 5–15s, tunable), randomly select a small number of devices and flip their `is_on` state.
- On every state change: update `last_changed` timestamp, recompute `current_power_w`.
- Expose a manual override function/endpoint so a device's state can be forced during a live demo (needed to reliably trigger alert conditions on camera).

### 3.2 Backend (FastAPI)
Single backend serving both clients. Responsibilities:
- Hold the in-memory list of 15 device objects (see §4).
- Run the **energy accumulator** loop: every tick, for each device currently `is_on`, add `current_power_w × elapsed_hours` to `total_energy_kwh_today`.
- Run the **alert engine**: every tick, check conditions (after-hours, >2hr continuous-on) and update the active alerts list.
- Serve REST endpoints (§5) for both dashboard and bot.
- Broadcast state diffs over WebSocket to connected dashboard clients.
- (Bonus) Push proactive alert messages to a Discord channel via a bot-side webhook/client when a new alert fires.

### 3.3 Web Dashboard (Next.js)
- Connects to backend via WebSocket for live push updates (no polling, no manual refresh).
- Renders:
  - Live device status panel, grouped by room.
  - Live power meter (total watts + per-room breakdown).
  - Active alerts panel (timestamped).
  - (Bonus) Top-view office layout with glowing lights / animated fans reflecting real state.

### 3.4 Discord Bot (discord.py)
- Calls the backend's REST API on each command (no local state).
- Formats raw JSON into a friendly sentence, optionally via an LLM call, before replying.
- (Bonus) Listens for alert events (via backend webhook or polling) and proactively posts to a designated channel.

## 4. Data Model

### Device object

| Field | Type | Example | Notes |
|---|---|---|---|
| `id` | string | `"work1-fan-1"` | Unique, stable identifier |
| `type` | enum | `"fan"` \| `"light"` | Determines rated wattage |
| `room` | enum | `"drawing"` \| `"work1"` \| `"work2"` | |
| `label` | string | `"Fan 1"` | Human-readable, per spec's `!status` format |
| `is_on` | boolean | `true` | Current state |
| `rated_power_w` | number | `60` | Fixed per type (fan=60W, light=15W) |
| `current_power_w` | number | `60` or `0` | Derived: `rated_power_w if is_on else 0` |
| `total_energy_kwh_today` | number | `0.42` | Accumulates while on; resets at day boundary |
| `last_changed` | ISO timestamp | `2026-07-04T14:32:00Z` | Updated on every state flip |
| `on_since` | ISO timestamp \| null | `2026-07-04T12:00:00Z` | Set when turned on, cleared when turned off — used for "on >2hrs" alert |

### Alert object

| Field | Type | Example |
|---|---|---|
| `id` | string | `"alert-001"` |
| `type` | enum | `"after_hours"` \| `"continuous_on"` |
| `room` | string | `"work2"` |
| `message` | string | `"Work Room 2 has been fully on for over 2 hours."` |
| `triggered_at` | ISO timestamp | `2026-07-04T22:00:00Z` |
| `active` | boolean | `true` |

## 5. Backend API

### REST endpoints

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/devices` | All 15 devices, current state |
| `GET` | `/rooms/{room}` | Devices for one room (`drawing`, `work1`, `work2`) |
| `GET` | `/usage` | Total current watts + total kWh today |
| `GET` | `/alerts` | Active alerts list |
| `POST` | `/devices/{id}/toggle` | Manual override — force a device on/off (demo + dashboard interactivity) |

### WebSocket

| Channel | Payload | Trigger |
|---|---|---|
| `/ws/devices` | Full or diffed device list + usage summary | Broadcast on every simulator tick / manual toggle |

### Suggested response shape (kept intentionally generic — not the final schema, just illustrative of structure)

```json
{
  "devices": [ { "id": "...", "room": "...", "is_on": true, "current_power_w": 60, "last_changed": "..." } ],
  "usage": { "total_watts_now": 740, "total_kwh_today": 4.2 },
  "alerts": [ { "type": "after_hours", "room": "work2", "message": "..." } ]
}
```

## 6. Discord Bot Commands

| Command | Backend call | Behavior |
|---|---|---|
| `!status` | `GET /devices` | Summarize all 3 rooms in one message |
| `!room <name>` | `GET /rooms/{name}` | Summarize one room |
| `!usage` | `GET /usage` | Report current watts + today's kWh |

All responses are built from live API data, then optionally passed through an LLM call to phrase conversationally (per spec's "humanized, friendly" requirement) — never hardcoded or randomly generated text.

---

## 7. Alert Logic

| Condition | Rule |
|---|---|
| **After-hours** | Any device with `is_on == true` and current time outside 09:00–17:00 |
| **Continuous-on (room level)** | All 5 devices in a room have been `is_on == true` continuously for > 2 hours (tracked via each device's `on_since`) |

Alerts are recomputed on each backend tick, timestamped when first triggered, and cleared when the condition no longer holds.

## 8. Tech Stack Summary

| Layer | Choice | Rationale |
|---|---|---|
| Backend | FastAPI (Python) | Async support for simulator/accumulator loops, native WebSocket support |
| Frontend | Next.js | Real-time UI, easy WebSocket client integration |
| Bot | discord.py | Standard, well-documented |
| State store | In-memory (Python dict/list) | No persistence requirement per spec; simplest for a hackathon timeline |
| Realtime transport | WebSocket | Satisfies "no manual refresh" requirement more cleanly than polling |
| LLM (bonus) | Claude/Gemini API | Humanizes bot responses |

## 10. Bonus Point Features

These are explicitly called out as bonus in the problem statement and worth attempting after the minimum-required features are solid:

1. **Animated office layout view** — top-view floor plan (provided in the spec) rendered on the dashboard, with lights visually glowing when ON and fan icons animating/spinning when running, instead of a plain list/table.
2. **Proactive Discord alerts** — the bot pushes a message to a designated channel automatically the moment an alert condition triggers (e.g., devices left on after hours), rather than only responding to manual commands.
3. **LLM-generated conversational bot replies** — using an LLM to phrase `!status` / `!room` / `!usage` responses in a natural, friendly tone instead of a raw data dump (explicitly "strongly encouraged" in the spec, and doubles as a stronger UX story for the demo video).

## 11. Open Decisions / Assumptions to Confirm

- **WebSocket vs. polling** for the dashboard — WebSocket recommended, but polling is an acceptable fallback if time is tight.
- **Simulator tick interval** — needs to be fast enough to look "live" in a 3-minute demo video, but not so fast it's chaotic on camera.
- **Manual toggle exposure** — recommended as both a demo aid (to reliably trigger alerts on camera) and a dashboard interactivity feature.
- **Day boundary for `total_energy_kwh_today` reset** — irrelevant for a short demo, but worth deciding (midnight local time is simplest) so the logic is at least documented even if never exercised live.
